## 概览

Bifrost 提供三个关键性能配置参数，用于控制吞吐量、内存使用和请求处理行为：

| 参数 | 作用范围 | 默认值 | 说明 |
|-----------|-------|---------|-------------|
| **并发数** | 每个 Provider | 1000 | 同时处理请求的 worker goroutine 数量 |
| **缓冲区大小** | 每个 Provider | 5000 | 在阻塞或丢弃前最多可排队的请求数 |
| **初始池大小** | 全局 | 5000 | sync pool 中预分配的对象数量，用于降低 GC 压力 |

<Info>
这些默认值适用于大多数最高约 5000 RPS 的生产部署。对于更高吞吐量或资源受限的环境，调优这些参数可以显著提升性能。
</Info>

## 理解参数

### 并发数（每个 Provider）

**作用：** 控制 Provider 性能的两个方面：

1. **Worker Goroutine：** 每个 Provider 用于处理请求的 goroutine 数量。每个 worker 会从 Provider 队列中取出请求，并调用对应 Provider 的 API。
2. **Provider 池预热：** 在 sync pool 中预分配 Provider 专用响应对象（例如 `AnthropicMessageResponse`、`OpenAIResponse`），减少请求处理期间的内存分配。

**影响：**

- **更高并发数** = 更多并行 Provider 请求、更高吞吐量、更多预分配响应对象
- **更低并发数** = 更少并行请求、更低资源使用，并更容易遵守 Provider 速率限制

**默认值：** 每个 Provider `1000` 个 worker

### 缓冲区大小（每个 Provider）

**作用：** 设置每个 Provider 的带缓冲 channel（队列）容量。传入请求会先进入此队列，随后由 worker 取出处理。

**影响：**

- **更大缓冲区** = 流量峰值期间可排队更多请求，更好处理突发流量
- **更小缓冲区** = 更低内存占用，更快向客户端施加背压

**默认值：** 每个 Provider 队列 `5000` 个请求

**队列满时行为：** 由 `drop_excess_requests` 控制：

- `false`（默认）：新请求会阻塞，直到队列有可用空间
- `true`：队列满时，新请求会立即被丢弃并返回错误

<Warning>
**约束：** 缓冲区大小必须大于或等于并发数。如果 `concurrency > buffer_size`，Provider 初始化会失败。
</Warning>

### 初始池大小（全局）

**作用：** 控制 Bifrost 启动时内部 sync pool 的预分配对象数量。这些池会复用对象，以降低垃圾回收开销。

**池化对象：**

- Channel message（请求包装对象）
- 响应 channel
- 错误 channel
- 流式 channel
- 插件 pipeline
- 请求对象

**影响：**

- **更高初始池大小** = 高流量下 GC 压力更低、延迟更稳定、初始内存使用更高
- **更低初始池大小** = 初始内存占用更低，但负载下可能产生更多分配

**默认值：** 每个池 `5000` 个对象

```json

        "initial_pool_size": 10000,
        "drop_excess_requests": false
    }
}
```

</Tab>

## 容量建议

### 并发数和缓冲区大小（每个 Provider）

根据特定 Provider 的预期 RPS，**按 Provider** 配置这些设置：

| Provider RPS | 并发数 | 缓冲区大小 |
|--------------|-------------|-------------|
| 100 | 100 | 150 |
| 500 | 500 | 750 |
| 1000 | 1000 | 1500 |
| 2500 | 2500 | 3750 |
| 5000 | 5000 | 7500 |
| 10000 | 10000 | 15000 |

<Info>
**示例：** 如果你预计 OpenAI 为 2000 RPS、Anthropic 为 500 RPS，则可将 OpenAI 配置为 `concurrency: 2000, buffer_size: 3000`，Anthropic 配置为 `concurrency: 500, buffer_size: 750`。
</Info>

**公式：**

```
concurrency = expected_rps
buffer_size = 1.5 × expected_rps
```

该比例可确保：

- 有足够队列容量吸收突发流量
- Worker 不会因为没有任务而空闲
- 在内存耗尽前施加背压

### 初始池大小（全局）

根据**所有 Provider 合计总 RPS**配置此设置：

| 总 RPS（所有 Provider） | 初始池大小 | 内存估算 |
|---------------------------|-------------------|-----------------|
| 100 | 150 | ~50 MB |
| 500 | 750 | ~100 MB |
| 1000 | 1500 | ~200 MB |
| 2500 | 3750 | ~400 MB |
| 5000 | 7500 | ~800 MB |
| 10000 | 15000 | ~1.5 GB |

<Note>
内存估算为近似值，会随请求/响应大小、Provider 数量和插件而变化。请监控你所在环境的实际内存使用。
</Note>

**公式：**

```
initial_pool_size = 1.5 × total_expected_rps
```

此外，确保：

```
initial_pool_size >= max(buffer_size across all providers)
```

这可确保池在运行时无需额外分配的情况下，预热到足以处理峰值队列深度。

## 多节点部署

当多个 Bifrost 实例位于负载均衡器后方运行时，应根据总预期 RPS，**将每节点设置除以节点数量**。

### 公式

```
Per-Node Concurrency = Total Concurrency / Number of Nodes
Per-Node Buffer Size = Total Buffer Size / Number of Nodes
Per-Node Initial Pool Size = Total Initial Pool Size / Number of Nodes
```

### 示例：4 个节点承载 10,000 RPS

**总容量（4 个节点合计）：**

- 总 RPS：10,000 RPS
- 单节点 RPS：每节点约 2,500 RPS

**单节点承载 10,000 RPS 时的设置：**

- 并发数：10000
- 缓冲区大小：15000
- 初始池大小：15000

**每节点设置（4 个节点，总计 10,000 RPS）：**

| 参数 | 总计（聚合） | 每节点（4 个节点） |
|-----------|-------------------|-------------------|
| 并发数 | 10000 | 2500 |
| 缓冲区大小 | 15000 | 3750 |
| 初始池大小 | 15000 | 3750 |

<Tab title="Go SDK">

```go
const numNodes = 4

func (a *MyAccount) GetConfigForProvider(provider schemas.ModelProvider) (*schemas.ProviderConfig, error) {
    // 总容量除以节点数量
    // 总计：4 个节点共 10,000 RPS = 每节点 2,500 RPS
    return &schemas.ProviderConfig{
        NetworkConfig: schemas.DefaultNetworkConfig,
        ConcurrencyAndBufferSize: schemas.ConcurrencyAndBufferSize{
            Concurrency: 10000 / numNodes, // 每节点 2500
            BufferSize:  15000 / numNodes, // 每节点 3750
        },
    }, nil
}

// 在 main 初始化中
bifrostConfig := schemas.BifrostConfig{
    Account:         myAccount,
    InitialPoolSize: 15000 / numNodes, // 每节点 3750
}
```

</Tab>
</Tabs>

<Tip>
**Kubernetes Horizontal Pod Autoscaling：** 使用 HPA 时，请按最小副本数配置这些设置。随着 Pod 扩容，每个节点会承载更小比例的流量。可以考虑使用环境变量或 ConfigMap，根据副本数动态调整设置。
</Tip>

## Provider 专项调优

不同 Provider 具有不同的速率限制和延迟特征。应分别调优每个 Provider：

### Provider 速率限制注意事项

| Provider | 典型速率限制 | 推荐并发数 | 说明 |
|----------|---------------------|------------------------|-------|
| OpenAI | 500-10000 RPM（因层级而异） | 100-500 | 更高层级支持更高并发 |
| Anthropic | 1000-4000 RPM（因层级而异） | 50-200 | 速率限制更保守 |
| Bedrock | 按模型限制 | 100-300 | 检查你的 AWS 账户配额 |
| Azure OpenAI | 按部署限制 | 100-500 | 按 deployment 配置 |
| Vertex AI | 按模型配额 | 100-300 | 检查 GCP 配额 |
| Groq | 极高吞吐量 | 500-1000 | 为高并发设计 |
| Ollama | 受本地资源约束 | 10-50 | 受本地 GPU/CPU 限制 |

## 队列溢出处理

当 Provider 队列达到容量上限时，Bifrost 的行为由 `drop_excess_requests` 控制：

### 阻塞模式（默认）

```json
{
    "config": {
        "drop_excess_requests": false
    }
}
```

- 新请求会**等待**，直到队列有可用空间
- 确保没有请求丢失
- 高负载期间可能增加延迟
- 适合每个请求都很重要的关键工作负载

### 丢弃模式

```json
{
    "config": {
        "drop_excess_requests": true
    }
}
```

- 队列满时，新请求会被**立即拒绝**
- 返回错误：`"request dropped: queue is full"`
- 保持已接受请求的延迟稳定
- 适合过期请求没有价值的实时应用

<Tip>
**最佳实践：** 在生产工作负载中使用 `drop_excess_requests: true`，并将缓冲区大小设置为并发数的 1.5 倍。这样既能处理合理的流量突发，又能防止内存耗尽。
</Tip>

## 监控和诊断

### 需要监控的关键指标

| 指标 | 健康范围 | 超出时的操作 |
|--------|---------------|-------------------|
| 队列深度 | < `buffer_size` 的 50% | 增加缓冲区或并发数 |
| 请求延迟（p99） | < 平均值的 2 倍 | 检查 Provider 速率限制 |
| 被丢弃请求 | 0 | 增加 `buffer_size` |
| 内存使用 | 稳定 | 降低池/缓冲区大小 |
| Goroutine 数量 | 稳定 | 检查 goroutine 泄漏 |

### 健康检查端点

Gateway 暴露健康检查和指标端点：

```bash
# 健康检查
curl http://localhost:8080/health

# Prometheus 指标
curl http://localhost:8080/metrics
```
